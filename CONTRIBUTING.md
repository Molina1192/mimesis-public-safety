# Contributing to Mimesis Public Safety

Thank you for your interest in contributing to the Mimesis Public Safety project.

This repository contains the official branding, graphics, documentation, and related assets for the Mimesis Roleplay public safety system. To ensure consistency and maintain quality, all contributions should follow the standards outlined below.

---

# Project Philosophy

This project is built around four core principles:

- Quality over speed.
- Originality over imitation.
- Consistency across all assets.
- Documentation alongside development.

Every contribution should support these principles.

---

# Getting Started

Before beginning work:

1. Read the project documentation.
2. Review any applicable standards.
3. Check existing issues and milestones.
4. Create or assign an issue before starting significant work.

---

# Development Workflow

Every feature follows the same lifecycle:

Research

↓

Planning

↓

Design / Development

↓

Review

↓

Approval

↓

Merge

↓

Release

---

# Branch Strategy

Do **not** work directly from the `main` branch.

Create a feature branch using one of the following formats:

```
feature/<name>
fix/<name>
docs/<name>
chore/<name>
```

Examples:

```
feature/lspd-badge
feature/city-crest
docs/update-roadmap
fix/fleet-numbering
```

---

# Commit Messages

Use clear, descriptive commit messages.

Examples:

```
feat: create City of Los Santos crest

feat: add LSPD badge artwork

docs: update project roadmap

fix: correct typography spacing

style: refine department color palette

chore: reorganize folder structure
```

---

# Asset Standards

All production assets must:

- Be original work.
- Follow established brand standards.
- Include editable source files.
- Be reviewed before release.
- Use approved naming conventions.
- Be stored in the correct directory.

---

# Documentation Requirements

Significant features should include documentation when appropriate.

Examples include:

- New branding systems
- Naming conventions
- Color palettes
- Typography standards
- Fleet standards

---

# Pull Requests

Pull requests should:

- Focus on one feature or issue.
- Include a clear description of the change.
- Reference the related issue when applicable.
- Be reviewed before merging.

---

# Code of Conduct

All contributors are expected to follow the project's Code of Conduct.

---

# Questions

If you are unsure about a contribution, open an issue or discuss it before beginning work.

Early communication helps prevent unnecessary rework.

---

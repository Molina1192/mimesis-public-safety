# Contributing to Mimesis Public Safety

Thank you for your interest in contributing to the **Mimesis Public Safety Project**.

This repository contains the official branding, graphics, documentation, and related Project Assets for the Mimesis Roleplay public safety identity system. To ensure consistency, quality, and long-term maintainability, all contributions must follow the standards outlined in this document.

---

## Before You Contribute

Before beginning any work, please review the following documents:

- [README.md](README.md)
- [LICENSE](LICENSE)
- [PROJECT_PRINCIPLES.md](PROJECT_PRINCIPLES.md)
- [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md)
- [SECURITY.md](SECURITY.md)

Understanding these documents will help ensure your contribution aligns with the project's goals and development standards.

---

## Project Principles

Every contribution should support the project's guiding principles:

- Quality over speed
- Originality over imitation
- Consistency across all Project Assets
- Documentation alongside development
- Long-term maintainability

When in doubt, prioritize quality and consistency over completing work quickly.

---

## Getting Started

Before starting work:

1. Review the relevant project documentation.
2. Check existing issues, milestones, or discussions.
3. Open or claim an issue before beginning significant work.
4. Discuss major changes with the Project Owner before implementation.

This helps prevent duplicate work and ensures development remains coordinated.

---

## Development Workflow

Every feature follows the same lifecycle:

```text
Research
    ↓
Planning
    ↓
Design / Development
    ↓
Review
    ↓
Approval
    ↓
Merge
    ↓
Release
```

No feature should bypass the review and approval process.

---

## Branch Strategy

Do **not** commit directly to the `main` branch.

Create a dedicated branch using one of the following naming conventions:

```text
feature/<name>
fix/<name>
docs/<name>
chore/<name>
```

Examples:

```text
feature/lspd-badge
feature/city-crest
docs/update-roadmap
fix/fleet-numbering
chore/repository-cleanup
```

---

## Commit Message Convention

Use concise, descriptive commit messages following a conventional format.

Examples:

```text
feat: create City of Los Santos crest
feat: add LSPD badge artwork
docs: update project roadmap
fix: correct typography spacing
style: refine department color palette
chore: reorganize folder structure
```

Meaningful commit history makes the project easier to review and maintain.

---

## Asset Standards

All Project Assets must:

- Be original work or properly licensed.
- Follow established brand standards.
- Include editable source files when applicable.
- Include production-ready exports when appropriate.
- Follow approved naming conventions.
- Be stored in the correct repository directory.
- Be reviewed and approved before release.

Assets that do not meet these standards may be rejected or returned for revision.

---

## Documentation Requirements

Documentation should accompany significant additions or changes whenever appropriate.

Examples include:

- Branding systems
- Department identities
- Color palettes
- Typography standards
- Fleet standards
- Naming conventions
- Design decisions

Documentation is considered part of the feature—not an optional extra.

---

## Pull Requests

Pull requests should:

- Focus on a single feature or issue.
- Include a clear summary of the proposed changes.
- Reference related issues when applicable.
- Pass any required project review.
- Remain focused and avoid unrelated modifications.

Large changes should be discussed before implementation whenever possible.

---

## Security

If your contribution involves a security concern or vulnerability, please follow the reporting process described in [SECURITY.md](SECURITY.md) instead of opening a public issue.

---

## Code of Conduct

All contributors are expected to follow the standards described in [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md).

Contributors who repeatedly violate project standards or community expectations may have their contributions declined or their access restricted.

---

## Questions

If you are unsure whether a proposed contribution is appropriate, please open a discussion or issue before beginning work.

Early communication helps maintain project quality, prevents unnecessary rework, and ensures that effort is directed toward the project's priorities.
